# CI Failure Atlas

CI Failure Atlas ingests CI failure data, builds semantic clustering outputs, and provides:

- a static triage site (weekly + global triage reports), and
- a local review app for human Phase3 linking.

## What This Project Does

The project combines ingestion, semantic processing, and reporting into one operational flow:

`controllers` -> `semantic phases (phase1/phase2/phase3)` -> `review + static triage site`

### 1) Ingestion via controllers

Controllers continuously ingest and derive facts into PostgreSQL facts/state tables:

- source ingestion (Sippy, Prow, GitHub),
- run/raw-failure fact materialization,
- daily metric rollups.

### 2) Semantic workflow

Semantic outputs are generated by a separate workflow/materialization task and persisted in PostgreSQL semantic snapshots:

- **Phase1**: normalize and classify failures into deterministic test-scoped clusters.
- **Phase2**: merge phase1 outputs into global failure signatures.
- **Phase3**: human-in-the-loop linking in the app review page to group equivalent signatures using durable row-level anchors.

The review app is the operator interface for Phase3 linking and cross-week reconciliation.

### 3) Static site for reporting

The static site is the reporting surface used for triage:

- weekly report (`weekly-metrics.html`)
- global signature triage (`global-signature-triage.html`)

## Prerequisites

- Go 1.25+
- Access to Sippy/Prow/GitHub APIs used by controllers
- (Optional, for publishing exported static HTML) Azure CLI authenticated to the target subscription

## Build and Run

### Run all controllers

This is the data ingestion and derived-facts pipeline.

```bash
go run cmd/main.go run \
  --source.envs dev,int,stg,prod \
  --history.weeks 4
```

Common options:

- `--storage.postgres.embedded.data-dir` (default: `data/postgres`)
- `--source.envs` (default: `dev`)
- `--history.weeks` (default: `4`)

### Materialize a semantic week

Builds one Sunday-starting semantic week in PostgreSQL from the current facts/state tables:

```bash
go run cmd/main.go semantic materialize \
  --week 2026-03-29
```

If `--week` is omitted, the command defaults to the current week start in UTC.

### Run the app

Serves the unified weekly/global/review app:

```bash
go run cmd/main.go app \
  --week 2026-03-15 \
  --app.listen 127.0.0.1:8082 \
  --history.weeks 4
```

### Export the static site

Exports static weekly/global/archive HTML from existing semantic snapshots in PostgreSQL:

```bash
go run cmd/main.go app export-site \
  --site.root site \
  --history.weeks 4
```

`app export-site` does not run semantic materialization. Run your separate semantic workflow task first, then export from the DB-backed snapshots.

### Publish the exported site

Use Azure CLI or another external script to upload the generated HTML. For example:

```bash
az storage blob upload-batch \
  --destination '$web' \
  --source site \
  --account-name <storage-account-name> \
  --auth-mode login \
  --overwrite
```

### Open the review page

The review UI is served from the unified app:

```bash
go run cmd/main.go app \
  --week 2026-03-15 \
  --history.weeks 4 \
  --app.listen 127.0.0.1:8082
```

Then open:

```bash
http://127.0.0.1:8082/review/
```

### Preview the exported site locally

Use any simple static file server. For example:

```bash
python -m http.server 8080 --directory site
```

Or:

```bash
npx serve site
```

## Typical Workflow

1. Run controllers (`cfa run`) to refresh facts/state in PostgreSQL.
2. Materialize semantic snapshots (`cfa semantic materialize`) for the week you want to review/export.
3. Open the app (`cfa app`) to review/link signatures (Phase3).
4. Export static HTML (`cfa app export-site`) when you want a storage-account-hosted snapshot.
5. Upload the exported `site/` directory with `az` or a small external script when ready to publish.
